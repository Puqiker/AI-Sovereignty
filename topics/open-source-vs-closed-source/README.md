# 开源 vs 闭源选择

> 这不是一个信仰问题——而是一个根据具体需求做出的工程决策。

---

## 决策矩阵

| 维度 | 开源 | 闭源 |
|------|------|------|
| 数据控制 | ✅ 完全自控 | ❌ 经过第三方 |
| 模型能力（当前） | 🟡 接近但有差距 | ✅ 最强 |
| 成本（低用量） | ❌ 硬件+运维成本 | ✅ 按量付费 |
| 成本（高用量） | ✅ 边际成本低 | ❌ 费用累积 |
| 定制化 | ✅ 可微调、可修改 | ❌ 有限 |
| 运维 | ❌ 需要自行运维 | ✅ 零运维 |
| 透明度 | ✅ 代码可审查 | ❌ 黑盒 |
| 供应商风险 | ✅ 低 | ❌ 高 |

---

## 典型选择策略

**选闭源API当**：
- 快速原型开发
- 低用量场景
- 需要最强模型能力
- 没有运维团队

**选开源自部署当**：
- 数据敏感不能出域
- 高用量（成本优化）
- 需要深度定制（微调）
- 需要完全控制和透明度

**混合策略**：
- 敏感数据用私有部署的开源模型
- 非敏感任务用闭源API
- 通过抽象层实现灵活切换
