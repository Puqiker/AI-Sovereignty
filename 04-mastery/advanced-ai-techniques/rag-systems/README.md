# RAG系统

> RAG（Retrieval-Augmented Generation，检索增强生成）解决了大语言模型的两个核心问题：
> 知识有截止日期、以及会幻觉。
> 它的思路很简单：在模型回答之前，先从可靠的外部数据源中检索相关信息，
> 把这些信息放进上下文，让模型基于这些信息来回答。

---

## 为什么需要RAG

> 📌 **前置知识**：模型的知识局限和幻觉成因。
> 参考 [幻觉分析](../../02-foundations/llm-internals/hallucination-analysis/)
> 和 [上下文窗口与记忆](../../02-foundations/llm-internals/context-window-memory/)。

大语言模型的知识有两个根本性的问题：

1. **知识过时**：训练数据有截止日期，之后的事情模型不知道
2. **知识不可靠**：模型的"知识"是参数中的统计模式压缩，可能不准确

RAG通过在推理时注入外部知识来解决这两个问题。

---

## RAG的完整流程
用户提问："公司最新的AI使用政策是什么？"

步骤1：检索（Retrieval）
把问题转换为向量 → 在知识库中搜索最相关的文档片段
找到：《AI使用政策v3.2》第3章、第5章

步骤2：增强（Augmentation）
把检索到的文档片段拼接到prompt中：
"基于以下参考资料回答用户问题：
[文档片段1]
[文档片段2]
用户问题：公司最新的AI使用政策是什么？"

步骤3：生成（Generation）
模型基于提供的参考资料生成回答
回答有据可查，大幅降低幻觉

## 核心技术组件

### 文档处理
原始文档（PDF、Word、网页、数据库……）
↓
[文档解析] → 提取纯文本
↓
[文本分块] → 把长文档切成适当大小的块（chunk）
↓
[向量化] → 用嵌入模型把每个块转换为向量
↓
[存储到向量数据库]


**分块策略**是RAG质量的关键因素之一：
- 块太大：包含太多不相关信息，稀释了相关信息的权重
- 块太小：上下文不完整，可能丢失关键信息
- 切割位置不当：在句子或段落中间切断，破坏语义完整性

### 向量检索

用户的问题也被转换为向量，
然后在向量数据库中找到距离最近（最相似）的文档块。

> 📌 **前置知识**：向量相似度。
> 参考 [线性代数 → AI](../../02-foundations/mathematics/linear-algebra/) 中的余弦相似度。

主流的向量数据库：Pinecone、Weaviate、Milvus、Chroma、FAISS等。

### 重排序（Reranking）

向量检索可能返回语义相关但并非最佳的结果。
重排序模型对检索结果进行二次排序，提升精准度。

## RAG的陷阱

### 检索质量是天花板

如果检索阶段没有找到正确的文档，模型不可能给出正确答案。
**RAG系统的质量上限由检索质量决定。**

### 上下文污染

如果检索到了不相关或矛盾的文档，模型可能被误导。
比检索不到更糟糕的是检索到了错误的文档——
模型会基于错误文档自信地给出错误答案。

### 不是所有问题都适合RAG

- 需要跨多个文档综合推理的问题：RAG检索到的是片段，可能缺乏全局视角
- 需要精确计算或逻辑推理的问题：RAG提供的是文本信息，不能替代推理能力
- 知识库中根本不存在答案的问题：RAG检索到的"最相关"内容可能和问题无关

---

## 关键认知

1. RAG = 检索 + 增强 + 生成——用外部知识弥补模型的知识局限
2. 分块策略和检索质量是RAG效果的核心决定因素
3. RAG降低幻觉但不能消除——检索错误或不相关文档可能引入新问题
4. 不是所有问题都适合RAG——需要判断问题类型和知识库覆盖范围
