# 问题拆解

---

## 从业务问题到技术方案

大多数业务问题不能直接扔给AI。你需要做的是：
模糊的业务需求
↓
[澄清问题] → 真正要解决的问题是什么？
↓
[可行性判断] → AI是否是解决这个问题的正确工具？
↓
[问题分解] → 拆成可以被AI处理的子问题
↓
[技术选型] → 每个子问题用什么AI技术
↓
[方案设计] → 整体架构和实现路径


### 澄清问题

业务方说"我们需要AI来提高效率"——这不是一个可以着手解决的问题。

你需要追问：
- 具体是哪个环节需要提高效率？
- 当前这个环节需要多少时间/人力？
- 什么样的结果算"效率提高了"？
- 错误的后果是什么？能接受多少错误率？

### 可行性判断

不是所有问题都适合用AI：

**适合AI的问题**：
- 有大量的相似任务需要处理
- 任务的输入和期望输出可以被明确定义
- 可以接受一定比例的错误（有兜底机制）
- 有足够的数据或示例

**不适合AI的问题**：
- 每个case都完全不同，没有模式可循
- 要求100%准确率，没有容错空间
- 涉及高度敏感的决策且无法人工审查
- 没有数据也没有示例

### 问题分解

一个复杂问题通常可以分解为多个子问题：
例：自动化客服工单处理

子问题1：工单分类（这个工单属于什么类型？）
子问题2：信息提取（工单中的关键信息是什么？）
子问题3：优先级判断（这个工单有多紧急？）
子问题4：回复生成（初步的回复建议是什么？）
子问题5：质量检查（生成的回复是否合适？）

每个子问题可以独立选择技术方案，然后组合成完整系统。

---

## 关键认知

1. 从模糊的业务需求到清晰的技术方案需要系统性的拆解过程
2. 先判断AI是否是正确的工具——不是所有问题都适合AI
3. 复杂问题应该分解为可独立处理的子问题
4. 每个子问题的错误容忍度决定了技术方案的选择
