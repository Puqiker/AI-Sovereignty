# AI工作流编排

> 复杂的AI应用不是单个模型调用，而是多个AI组件的协作。
> 工作流编排就是设计和管理这些组件之间的协作方式。

---

## 编排的核心问题

### 1. 数据流设计

每个组件的输出格式必须和下一个组件的输入格式匹配。

组件A输出JSON → 组件B期望纯文本 → 需要格式转换层


### 2. 错误处理

> 📌 **安全关联**：错误传递与放大。
> 参考 [AI工作流威胁](../../03-security/threat-landscape/ai-workflow-threats/)。

每个组件都可能失败。编排层需要处理：
- 组件超时
- 组件返回异常结果
- 组件返回低置信度结果

处理策略：
- **重试**：对临时性错误有效
- **降级**：使用备用组件或简化方案
- **中断**：停止工作流，报告错误，请求人工介入
- **跳过**：某些非关键步骤可以跳过

### 3. 可观测性

在生产环境中你需要知道工作流的运行状态：
- 每个组件的执行时间
- 每个组件的成功/失败率
- 端到端的延迟
- 关键指标的趋势

### 编排工具

- **LangChain / LangGraph**：Python生态中最流行的LLM编排框架
- **LlamaIndex**：专注于RAG场景的编排
- **自建编排**：对于简单工作流，直接用Python代码编排可能比框架更清晰

选择框架还是自建的判断：
- 工作流简单（3步以内）→ 自建代码，避免框架的学习和维护成本
- 工作流复杂（多分支、多组件、需要记忆）→ 考虑使用框架

---

## 关键认知

1. 工作流编排 = 数据流设计 + 错误处理 + 可观测性
2. 每两个组件之间需要格式转换和验证
3. 为每个可能的失败点设计处理策略——不要假设每一步都会成功
4. 简单工作流用代码，复杂工作流考虑框架
