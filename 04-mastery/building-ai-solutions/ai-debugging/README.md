# AI调试

> 传统软件的bug是确定性的——同样的输入总是产生同样的错误输出。
> AI系统的"bug"是概率性的——同样的输入可能有时对有时错。
> 这需要一种不同的调试思维。

---

## 诊断框架

当AI系统给出了错误或不理想的输出，按以下层次排查：
第1层：输入问题
- 输入数据是否有质量问题？
- Prompt设计是否清晰明确？
- 输入是否超出了模型的能力范围？
- 是否有Prompt注入？

第2层：检索问题（如果使用RAG）
- 检索到的文档是否相关？
- 分块策略是否合理？
- 嵌入模型的质量是否足够？

第3层：模型问题
- 是否属于模型的已知弱点？（数学、最新事实等）
- Temperature和采样参数是否合适？
- 上下文窗口是否溢出？
- 是否是幻觉？

第4层：后处理问题
- 输出解析是否正确？
- 格式转换是否丢失了信息？
- 过滤规则是否误杀了正确输出？

第5层：系统问题
- API调用是否有超时或限流？
- 缓存是否返回了过期结果？
- 并发是否导致了资源竞争？


### 调试的概率性思维

传统调试：找到那一个bug → 修复 → 问题消失。

AI调试：
- 同一个问题可能有多个原因
- "修复"可能只是降低了出错概率，不是彻底消除
- 需要用统计方法评估——在多少个测试case上表现改善了

### 最小复现

和传统调试一样，找到**最小的输入**能触发问题：
- 简化prompt，看哪些部分是触发问题的关键
- 简化上下文，看信息长度是否是问题所在
- 改变采样参数，看是否是随机性导致的

---

## 关键认知

1. AI调试是分层的——从输入到检索到模型到后处理到系统逐层排查
2. AI的问题是概率性的——"修复"意味着降低出错概率而非消除
3. 用统计方法评估修复效果——不是看单个case，是看一批case的表现变化
4. 最小复现帮你定位问题的核心触发条件
