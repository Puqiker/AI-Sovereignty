# AI供应链攻击

> 你使用的AI系统不是从零构建的——
> 你用的是别人训练的模型、别人标注的数据集、别人写的推理框架、
> 别人发布的微调适配器、别人封装的插件和skills。
> 供应链上的每一环都可能被攻击者渗透。

---

## AI供应链的构成
你的AI应用
↑
[应用框架 / 编排工具] ← LangChain, LlamaIndex等
↑
[推理引擎] ← vLLM, llama.cpp等
↑
[模型权重] ← 从HuggingFace等平台下载
↑
[微调数据集] ← 可能来自第三方
↑
[预训练数据] ← 通常不可控
↑
[基础依赖] ← PyTorch, CUDA, Python包等


**每一层都是潜在的攻击面。**

---

## 攻击方式

### 模型权重篡改

攻击者修改开源模型的权重文件并重新发布：
- 在模型中植入后门行为
- 修改模型使其在特定输入下输出恶意内容
- 甚至在权重文件中嵌入可执行的恶意代码

模型权重文件通常使用pickle格式（Python），
pickle在反序列化时可以执行任意代码。
**下载并加载一个不可信的模型权重文件，等同于运行不可信的可执行文件。**

### 恶意插件/Skills

你之前在项目设计中提到过的问题——
别人封装好的skills你拿来直接用，你不知道里面是否有：
- 隐藏的prompt注入
- 数据外传行为（把你的输入发送到第三方）
- 误导性的输出操纵

### 依赖包攻击

AI项目依赖大量的Python包。
攻击者可以：
- 发布和知名包名称相似的恶意包（typosquatting）
- 入侵已有包的维护者账号，在更新中注入恶意代码
- 在依赖链的深处注入恶意代码（你可能不知道你间接依赖了哪些包）

### 数据集投毒（跨供应链）

公开的数据集（HuggingFace Datasets等）可能被恶意修改。
如果你使用这些数据集做微调，投毒的数据就会影响你的模型。

## 防御方案

### 模型来源验证

- 只从官方渠道下载模型权重
- 验证文件的哈希值/数字签名
- 优先使用safetensors格式（不允许执行任意代码）而非pickle
- 在隔离环境中首先加载和测试下载的模型

### 依赖审计

- 使用依赖锁定（pip freeze / poetry.lock）固定版本
- 定期扫描依赖中的已知漏洞（如使用safety、pip-audit等工具）
- 审查新引入的依赖——特别是不知名的包

### 插件/Skills安全

- 审查第三方插件的源代码
- 在沙箱环境中测试插件行为
- 监控插件的网络请求——是否在向未预期的地址发送数据
- 正如你在项目设计中提出的：**拿到别人的上下文然后自行修改，
  而不是直接执行别人封装好的黑盒**

### 供应链安全体系

- 建立AI系统的**软件物料清单（AI-SBOM）**——记录所有组件的来源、版本、哈希
- 对供应链中的每一环进行风险评估
- 建立供应商安全评估流程

---

## 关键认知

1. AI供应链涵盖模型权重、数据集、推理引擎、依赖包、插件等多个层面
2. pickle格式的模型权重可以执行任意代码——下载不可信模型等同运行恶意程序
3. 恶意插件/skills可能包含prompt注入、数据外传、输出操纵
4. 防御需要来源验证、依赖审计、插件审查、以及完整的供应链安全体系
5. AI-SBOM是追踪和管理供应链风险的关键工具
