# 基于原理的Prompt Engineering

> 大多数Prompt Engineering教程教你背诵技巧："把指令放在开头""使用分步骤格式"。
> 但它们不告诉你为什么。
> 你现在已经理解了模型的工作原理，
> 这个模块从原理出发推导出Prompt设计的逻辑——
> 让你知其然更知其所以然。

---

## 核心思维转变

从"背技巧"转变为"理解模型在做什么，然后推导出最佳输入方式"。

你现在知道：
- 模型在做条件概率预测：P(下一个token | 前面所有token)
- 模型通过注意力机制从前文中提取相关信息
- 注意力有位置偏好——开头和结尾信息更容易被注意到
- 模型是一个token一个token生成的——前面的输出影响后面的输出
- 模型的"知识"是训练数据中模式的压缩

**每一条Prompt设计原则都可以从上面这些事实中推导出来。**

---

## 原则一：你的输入决定了概率分布的起点

### 原理

模型计算的是 P(输出 | 输入)。
不同的输入导致完全不同的概率分布。

同一个问题，不同的表述方式，
激活的是模型参数中不同的模式。

### 推论

**清晰、明确的输入 → 更集中的概率分布 → 更可靠的输出。**

模糊的输入："说说AI"
→ 概率分布很分散——可以接"的发展历史""的技术原理""的应用场景"……
→ 模型可能给出你不想要的方向

明确的输入："解释Transformer中自注意力机制的Query-Key-Value计算过程"
→ 概率分布很集中——几乎确定地会输出关于QKV计算的技术解释
→ 输出更可能是你想要的


这不是"要写清楚问题"这种常识建议。
这是一个数学事实：你的输入决定了概率分布的形状。

---

## 原则二：位置影响注意力

### 原理

你在上下文窗口模块学到了"中间遗失"效应：
模型对输入开头和结尾的信息给予更高的注意力权重。

### 推论

**把最重要的信息放在输入的开头或结尾。**

如果你有一段很长的背景资料加一个问题：
效果较差的结构：
[问题]
[大段背景资料]

效果较好的结构：
[简短的任务说明] ← 开头：模型注意力高
[背景资料] ← 中间：注意力相对低
[具体问题和要求] ← 结尾：模型注意力高


这不是"排版美观"的建议。
这是基于注意力机制的位置敏感性做出的结构性优化。

---

## 原则三：分步骤指令利用了自回归特性

### 原理

模型是一个token一个token生成的。
每一步的输出成为下一步的输入的一部分。

如果你让模型"先想后说"——先输出推理步骤，再输出结论——
那么在生成结论时，前面输出的推理步骤已经在上下文中了，
模型的注意力可以参考这些步骤。

### 推论

**思维链（Chain of Thought）有效的本质原因：
它把中间推理步骤变成了上下文中的显式信息，
让模型在生成后续内容时可以参考这些中间结果。**

直接回答："答案是42。"
→ 模型在一步内要完成整个推理，容易出错

思维链："第一步，我们知道……第二步，根据……第三步，因此……答案是42。"
→ 每一步的推理结果都写在上下文中，后续步骤可以参考前面的步骤
→ 复杂推理被分解为多个简单步骤，每一步出错的概率降低


这不是"让AI慢慢想"的心理学建议。
这是利用了自回归生成的机制——前面的输出成为后面的输入。

---

## 原则四：Few-shot示例改变概率分布

### 原理

你在概率统计模块学过条件概率。
Few-shot示例做的事情是：**改变了条件，从而改变了概率分布。**

当你给模型一些示例：

输入：Happy → 输出：开心
输入：Sad → 输出：悲伤
输入：Angry → 输出：

模型看到这个前文时，P("愤怒" | 前文) 远高于 P("angry是一个形容词" | 前文)。

因为前面的示例建立了一个明确的模式——"英语 → 中文翻译"，
模型的概率分布被这个模式强烈地塑造了。

### 推论

**Few-shot示例的本质是在上下文中建立一个局部的概率分布偏置。**

- 示例越一致（都遵循同一个模式），偏置越强，模型越倾向于遵循这个模式
- 示例越多，模式越明确——但太多也有边际递减（注意力的稀释效应）
- 示例的质量比数量重要——一个高质量的示例好过三个不一致的示例

## 原则五：System Prompt设置全局条件

### 原理

System Prompt（系统提示）出现在对话的最开始，
它设定了整个对话的基础条件。

从概率的角度：后续每一步的生成都是在System Prompt存在的条件下计算的。
P(回答 | System Prompt + 用户输入) ≠ P(回答 | 用户输入)


System Prompt通过占据上下文的开头位置（注意力高），
持续影响所有后续生成的概率分布。

### 推论

**System Prompt适合放那些需要全局生效的指令：**
- 角色设定："你是一个资深的安全工程师"
- 行为约束："只基于提供的文档回答，如果文档中没有相关信息，说明你不知道"
- 输出格式要求："使用Markdown格式，包含代码块"

不适合放在System Prompt中的内容：
- 具体的一次性问题
- 经常变化的上下文信息

## 原则六：Token预算意识

### 原理

上下文窗口是有限的。
输入越长，留给输出的空间越少。
而且注意力在长序列上的精度下降。

### 推论

**你需要有Token预算意识——精选最相关的信息放入上下文。**

实际操作：
- 不要把整个文档塞进去——提取最相关的段落
- 不要保留对话中所有的历史——只保留和当前问题相关的历史
- 在复杂任务中，考虑分步骤处理——每一步只处理一部分问题，减少单次上下文的负担

---

## 原则七：不同交互目的选择不同方式

### 原理

模型对不同类型的输入激活不同的概率分布。

### 推论

**不是所有交互都适合结构化指令。**

| 交互目的 | 适合的输入方式 | 原因 |
|---------|-------------|------|
| 执行具体任务 | 结构化分步指令 | 减少歧义，引导模型进入执行模式 |
| 探索性讨论 | 自然语言表达 | 保持语境完整性，不过早限制方向 |
| 创意生成 | 开放性描述 + 高Temperature | 允许模型探索更多可能性 |
| 精确回答 | 明确问题 + 低Temperature | 减少随机性，聚焦于最可能的答案 |
| 格式化输出 | Few-shot示例 | 示例比文字描述更能精确定义格式 |

## 原则八：当Prompt不工作时的诊断思路

### 基于原理的诊断

当模型给出不理想的输出时，不要只是换个说法重试。
用你对原理的理解来诊断问题：

1. **输出偏离主题？**
   → 检查你的输入是否足够明确。模糊的输入导致分散的概率分布。

2. **忽略了你提供的重要信息？**
   → 检查这些信息是否在输入的中间位置（中间遗失效应）。
   → 考虑将重要信息移到开头或结尾。

3. **推理步骤出错？**
   → 检查是否要求模型一步到位地完成复杂推理。
   → 考虑使用思维链，让模型分步骤输出。

4. **输出格式不对？**
   → 文字描述格式可能不够精确。
   → 考虑用Few-shot示例来定义格式。

5. **编造事实？**
   → 检查问题是否超出了模型训练数据的覆盖范围。
   → 考虑使用RAG提供可靠的参考文档。
   → 降低Temperature。

6. **输出被截断？**
   → 可能超出了上下文窗口的输出空间。
   → 精简输入或分多次请求。

## 从技巧到直觉

当你基于原理来设计Prompt时，
你不再需要记忆具体的技巧清单。

你需要的思维过程是：
1. 我想让模型做什么？
2. 模型要完成这件事需要什么信息？
3. 这些信息应该以什么结构组织？（考虑位置效应）
4. 模型在生成时可能遇到什么困难？（考虑任务复杂度）
5. 如何通过输入设计来降低这些困难？（考虑思维链、示例、约束）
6. 结果是否需要高准确性还是高创造性？（设置Temperature等参数）


**这个思维过程是通用的——它不依赖于任何特定的AI工具。**
不管你用ChatGPT、Claude还是未来的任何工具，
这个基于原理的思维方式都适用。

---

## 关键认知

1. Prompt设计的每一条原则都可以从模型工作原理中推导出来
2. 输入的清晰度决定概率分布的集中度——模糊输入导致分散输出
3. 位置效应是注意力机制的数学性质——重要信息放在开头或结尾
4. 思维链利用了自回归特性——前面的输出成为后面的上下文
5. Few-shot示例通过建立模式来偏置概率分布
6. 不同交互目的需要不同的输入方式——不是所有场景都适合结构化指令
7. 诊断Prompt问题时，用原理来定位原因，而不是盲目重试
