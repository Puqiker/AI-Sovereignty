# 推理机制

> "推理"在这里指的不是"逻辑推理"，
> 而是模型在部署后处理用户输入并生成输出的过程——也叫"推断"（inference）。
> 理解这个过程让你能理解模型响应速度的瓶颈、
> 为什么生成长文本比短文本慢、以及各种生成参数的实际含义。

---

## 自回归生成：一次一个token

你已经多次听到这个事实，现在看看它在推理时的具体含义。

当你输入"请解释量子计算"，模型的响应过程是：
第1步：
输入：[请, 解释, 量子, 计算]（4个token）
模型处理所有4个token，输出第5个token的概率分布
采样得到："量"

第2步：
输入：[请, 解释, 量子, 计算, 量]（5个token）
模型处理所有5个token，输出第6个token的概率分布
采样得到："子"

第3步：
输入：[请, 解释, 量子, 计算, 量, 子]（6个token）
...

（重复直到生成结束标记或达到最大长度）


**每一步只生成一个token。**
生成一个100 token的回答需要跑100次模型。

这就是为什么你在使用AI时能看到文字一个一个蹦出来——
模型确实是在一个一个地生成。

---

## KV Cache：避免重复计算

你注意到上面的过程有一个巨大的浪费：

第2步的输入是 [请, 解释, 量子, 计算, 量]。
但 [请, 解释, 量子, 计算] 这4个token在第1步已经处理过了。
第2步只多了一个token "量"。

如果每一步都从头处理整个序列，计算量随着生成的token数线性增长——
生成第100个token时要处理前面99个token，生成第1000个token时要处理前面999个token。

**KV Cache** 解决了这个问题。

在注意力计算中，每个位置需要的是 Key 和 Value 向量。
KV Cache 的做法是：**每一步计算完之后，把新token的K和V存起来。**
下一步只需要计算新token的Q、K、V，然后和之前缓存的K、V拼接做注意力。
第1步：处理4个token，缓存4组KV
第2步：只处理第5个token，用它的Q和缓存的4+1组KV做注意力
第3步：只处理第6个token，用它的Q和缓存的5+1组KV做注意力
...


每一步只做一个新token的计算，而不是重新处理整个序列。

**KV Cache是推理效率的关键优化。** 没有它，长文本生成会慢得不可用。

但KV Cache有一个代价：**它需要内存来存储。**
对于大模型和长序列，KV Cache可能占用几十GB的内存。
这也是上下文窗口长度在实际中受限的原因之一。

## 生成策略：从概率分布到具体文字

模型每一步输出的是整个词表上的概率分布。
怎么从这个分布中选出实际的token？这就是**解码策略**。

### 贪心解码（Greedy Decoding）

每一步选概率最高的token。

P("量") = 0.35, P("它") = 0.20, P("这") = 0.15, ...
→ 选择"量"

优点：确定性，输出可复现。
缺点：倾向于生成"安全"但无聊的文本。而且可能陷入重复循环。

### Temperature采样

在softmax之前把logits除以温度参数T：
P = softmax(logits / T)

- T = 1：标准分布
- T < 1：分布变尖锐——更确定，更倾向于选高概率token
- T > 1：分布变平坦——更随机，更可能选低概率的"意外"token

**Temperature控制的是创造力和准确性之间的权衡。**
写诗用高温度（更有创意），做数学题用低温度（更精确）。

### Top-k采样

只从概率最高的k个token中采样，忽略其余。

k = 10：只考虑概率最高的10个token，重新归一化后采样

避免了从概率极低的token中采样的风险（那些token通常是荒谬的选择）。

### Top-p（核采样, Nucleus Sampling）

从概率从高到低累加直到总概率达到p的最小token集合中采样。

p = 0.9：选择概率最高的若干token，使得它们的概率之和≥0.9


比top-k更灵活——当模型很确定时（概率集中在少数token上），
候选集很小；当模型不确定时，候选集自动变大。

### 重复惩罚

对已经出现过的token施加惩罚——降低它们在后续生成中被选中的概率。
防止模型陷入"今天天气好天气好天气好……"的循环。

### 这些参数的实际影响

当你在使用AI工具时调整这些参数，你实际上在控制的是：

| 参数 | 低值 | 高值 |
|------|------|------|
| Temperature | 更确定、更保守 | 更随机、更有创意 |
| Top-k | 限制更多、更安全 | 限制更少、更多样 |
| Top-p | 候选更少、更聚焦 | 候选更多、更发散 |

## Prefill vs Decode：两个阶段

推理实际上分为两个不同的计算阶段：

### Prefill阶段

处理用户输入的所有token。
这些token是一次性全部给定的，可以**并行处理**。

用户输入 [请, 解释, 量子, 计算]
→ 4个token同时通过Transformer
→ 生成KV Cache
→ 输出第5个token的概率分布


这个阶段是计算密集型的——主要瓶颈是GPU算力。

### Decode阶段

一个token一个token地生成输出。
每一步**顺序执行**，不能并行（因为每一步的输入依赖前一步的输出）。

生成 token 5 → 生成 token 6 → 生成 token 7 → ...


这个阶段是内存带宽密集型的——
每一步都需要从内存中读取整个模型的参数和KV Cache，
但只做很少的计算（只处理一个token）。

**这就是为什么AI回答时第一个字出来比较慢（prefill阶段），
后面的字一个一个跳出来的速度相对稳定（decode阶段）。**

## 推理成本

推理的成本取决于：

- **模型大小**：参数越多，每一步的计算和内存开销越大
- **输入长度**：输入越长，prefill阶段越慢，KV Cache越大
- **输出长度**：输出越长，decode阶段执行的步数越多
- **并发用户数**：同时服务多个用户时，GPU内存需要容纳多份KV Cache

**这就是为什么AI API按token收费——每一个token都需要实际的计算资源。**
通常输入token和输出token的单价不同——输出token更贵，
因为decode阶段的效率低于prefill阶段。

---

## 关键认知

1. 推理是自回归的——每次只生成一个token，需要多次模型运行
2. KV Cache避免了重复计算——用内存换速度
3. Temperature、Top-k、Top-p控制生成的确定性和多样性
4. Prefill阶段并行处理输入（算力密集），Decode阶段逐步生成输出（带宽密集）
5. 推理成本与模型大小、输入长度、输出长度直接相关
6. 理解这些让你能判断为什么某些操作慢、为什么某些参数影响输出风格
