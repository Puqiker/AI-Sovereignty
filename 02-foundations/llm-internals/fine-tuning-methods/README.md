# 微调方法

> 预训练给了模型广泛的语言能力。
> 微调把这种广泛的能力聚焦到特定的任务或行为模式上。
> 如果预训练是"通识教育"，微调就是"专业培训"。

---

## 为什么需要微调

预训练模型的问题在上一模块已经说过——
它只会"续写"，不会"对话"。

给一个纯预训练模型输入"请解释什么是量子计算"，
它可能输出：
"请解释什么是量子计算的基本原理，以及它与经典计算的区别。
请解释什么是量子纠缠。
请解释什么是……"


它在续写"请解释什么是XX"这个模式，而不是在回答你的问题。

微调的目标是：让模型学会一种新的行为模式——
理解用户的问题，然后给出有用的回答。

---

## 监督微调（SFT：Supervised Fine-Tuning）

### 方法

准备一批高质量的（指令, 回答）对：
指令："用简单的语言解释什么是机器学习"
回答："机器学习是让计算机从数据中自动发现规律的技术。
和传统编程不同，传统编程需要人写规则，
而机器学习让计算机自己从例子中学习规则……"

指令："写一首关于秋天的诗"
回答："枫叶染红山间路，
秋风送爽入人家。
……"


然后用这些数据继续训练预训练模型——
和预训练一样做"预测下一个token"，
但训练数据从互联网文本变成了精心编写的指令-回答对。

### SFT的效果

经过SFT之后，模型学会了：
- 识别"指令"模式并给出相应的"回答"
- 输出的格式更像一个助手的回答而不是续写
- 对不同类型的指令（问答、写作、分析、编程）给出不同风格的回答

### SFT的局限

SFT数据的数量远远少于预训练数据——
通常只有几万到几十万条高质量对话，
而预训练数据有万亿token。

SFT不是在教模型新的知识（知识主要来自预训练），
而是在教模型一种新的"行为模式"——用对话的方式来输出它已有的知识。

## 全量微调 vs 参数高效微调

### 全量微调（Full Fine-Tuning）

更新模型的所有参数。

优点：效果最好，模型可以充分适配新任务。
缺点：需要和预训练几乎相同的计算资源，
      而且会修改所有参数——如果训练数据量太小，容易过拟合，
      模型可能"忘记"预训练学到的部分知识（灾难性遗忘）。

### 参数高效微调（PEFT）

只更新模型的一小部分参数，其余参数冻结不动。

**LoRA（Low-Rank Adaptation）**——当前最流行的PEFT方法：

核心思想来自数学基础线性代数模块的知识——低秩近似。

在微调时，不直接修改原始的权重矩阵 W（维度可能是 4096×4096），
而是在旁边加一个低秩的"旁路"：
原始：y = Wx
LoRA：y = Wx + BAx

其中：
W：原始权重矩阵，4096×4096，冻结不动
A：4096×r 的矩阵（r远小于4096，比如r=8或r=16）
B：r×4096 的矩阵
BA 的秩最多为 r


只训练 A 和 B，参数量从 4096×4096 = 1600万 降到 4096×8 + 8×4096 = 65536——
只有原来的0.4%。

**为什么这样有效？**

研究发现，微调过程中参数的变化量通常集中在一个低秩子空间中——
也就是说，虽然权重矩阵很大，但微调需要改变的"方向"并不多。
LoRA就是只学习这些少数的关键方向。

### LoRA的实际优势

- **存储高效**：原始模型保持不变，只需要存储小的A和B矩阵。多个LoRA适配器可以共享同一个基础模型——每个适配器只有几十MB，而基础模型可能有几十GB。

- **切换灵活**：针对不同任务训练不同的LoRA适配器，使用时动态加载对应的适配器即可。一个基础模型可以服务多个任务。

- **降低门槛**：全量微调一个70B参数的模型可能需要几十张高端GPU。用LoRA微调，一张消费级GPU可能就够了。

## 指令微调（Instruction Tuning）

指令微调是SFT的一个特定形态——
训练数据是多种多样的指令和对应的回答。

关键在于指令的**多样性**：
问答类："太阳系中最大的行星是什么？" → "木星是太阳系中最大的行星……"
创作类："写一段关于AI的科幻故事开头" → "2045年，当最后一个人类程序员……"
分析类："分析这段代码的安全漏洞" → "这段代码存在SQL注入风险，因为……"
翻译类："把这段话翻译成英语" → "..."
摘要类："总结以下文章的要点" → "..."

指令微调让模型学会"遵循指令"这一通用能力——
不是学会回答某一类具体的问题，
而是学会"用户给我什么指令我就尝试执行什么"的行为模式。

## 微调的关键决策

当你（或你的公司）需要微调一个模型时，需要做以下判断：

### 1. 是否需要微调

不是所有问题都需要微调。
如果你的需求可以通过精心设计的Prompt在通用模型上解决，
那就不需要微调——微调有成本、有风险、需要维护。

需要微调的典型场景：
- 模型需要在特定领域（法律、医学）中有显著更好的表现
- 需要模型遵循特定的输出格式
- 需要将公司特有的知识内化到模型中
- 现有模型在你的场景下的表现不够好，且Prompt优化已经到达瓶颈

### 2. 全量微调还是LoRA

- 资源充足、数据量大、需要最大性能改变 → 全量微调
- 资源有限、需要快速迭代、需要多任务适配 → LoRA

### 3. 微调数据的质量

微调数据的质量比数量重要得多。

1000条高质量的、精心编写的指令-回答对，
效果可能好于10万条随意生成的低质量数据。

数据中的错误和偏差会被模型学习——
一条有错误的训练数据可能影响模型在相关场景下的所有输出。

---

## 关键认知

1. 微调把通用模型变成特定任务的专家——不是教新知识，而是教新行为
2. SFT用精心编写的指令-回答对训练模型学会"对话模式"
3. 全量微调效果最好但成本高，LoRA通过低秩近似大幅降低成本
4. LoRA的数学基础是低秩矩阵分解——微调的变化集中在少数关键方向
5. 指令微调让模型学会"遵循指令"这一通用能力
6. 微调数据的质量比数量重要——少量高质量数据优于大量低质量数据

## 下一步

微调让模型学会了对话和遵循指令。
但一个会对话的模型不一定是一个"好"的对话者——
它可能啰嗦、可能编造、可能在该拒绝的时候不拒绝。
让模型的行为符合人类偏好，需要RLHF对齐。
