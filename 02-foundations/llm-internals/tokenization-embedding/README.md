# 分词与嵌入

> 在觉醒阶段你已经知道了：模型看到的不是文字，而是数字。
> 这个模块详细解释这个转换是怎么发生的。
> 分词和嵌入是模型处理语言的第一步——如果这一步出了问题，后面一切都白费。

---

## 分词（Tokenization）：文字怎么切成片段

### 为什么不用字符或单词

两个极端的方案：

**方案一：按字符切分。**
"今天天气很好" → ["今", "天", "天", "气", "很", "好"]

问题：序列太长。一个几百字的文章变成几百个token，
计算量随序列长度平方增长（注意力的O(n²)复杂度），效率低下。
而且单个字符携带的语义信息太少，模型需要更多层来组合字符理解含义。

**方案二：按词切分。**
"I love natural language processing" → ["I", "love", "natural", "language", "processing"]

问题：词表太大。英语有几十万个词，加上各种变形（running, ran, runs）、
专业术语、新造词，词表膨胀到无法管理。
而且遇到从未见过的新词（out-of-vocabulary）就完全无法处理。

### 实际方案：子词分词（Subword Tokenization）

现代大语言模型使用的是介于字符和词之间的方案——**子词（subword）**。

核心思想：常见的词保持完整，不常见的词拆成更小的有意义的片段。

"understanding" → ["under", "stand", "ing"]
"unhappiness" → ["un", "happiness"] 或 ["un", "happ", "iness"]
"今天" → ["今天"] （常见词，保持完整）
"量子纠缠" → ["量子", "纠", "缠"] （不太常见，拆开）

### BPE算法（Byte Pair Encoding）

当前最主流的子词分词算法。GPT系列、Llama等都使用BPE或其变体。

BPE的训练过程：

1. 从所有单个字符开始作为初始词表
2. 统计训练语料中所有相邻token对的出现频率
3. 把频率最高的token对合并成一个新token，加入词表
4. 重复步骤2-3，直到词表达到预设大小（比如50000）

例子（简化）：
语料："aab aab ab"

初始词表：{a, b}
最频繁的对：a+a → 合并为 "aa"
词表变成：{a, b, aa}

再统计：最频繁的对：aa+b → 合并为 "aab"
词表变成：{a, b, aa, aab}

继续直到词表大小足够...

实际的BPE词表通常有32000到100000个token。

### 分词的影响

分词方式直接影响模型性能：

- **不同语言的效率差异**。英语一个词通常对应1-2个token，中文一个字通常对应1-2个token，但有些语言（如日语、阿拉伯语）可能需要更多token来表示同样长度的内容。这意味着同样的上下文窗口大小，对不同语言的实际容量不同。

- **数字和代码**。数字经常被拆得很碎——"123456"可能变成["123", "456"]或更细。这是大模型做数学不太可靠的原因之一——数字在分词阶段就被打散了，模型要从碎片中重构数字的含义。

- **罕见词的表示**。越罕见的词被拆得越碎，模型处理它需要更多token，效率更低。

## 嵌入（Embedding）：数字怎么变成含义

### Token ID

分词之后，每个token被映射为一个整数ID：
词表：{"今天": 4523, "天气": 8901, "很": 2345, "好": 7890, ...}

"今天天气很好" → [4523, 8901, 2345, 7890]

但整数ID本身不包含语义信息——4523和4524在数值上只差1，
但它们代表的词可能完全无关。

### 嵌入矩阵

嵌入矩阵是一个大小为 (词表大小 × 嵌入维度) 的矩阵。
词表大小：50000
嵌入维度：768（或4096，取决于模型）

嵌入矩阵 E：50000 × 768

查找嵌入就是用token ID作为索引，取出对应的行：
token "今天"，ID = 4523
嵌入向量 = E[4523] = [0.23, -0.87, 1.45, ..., -0.56] （768个数字）

### 嵌入向量的语义

你在数学基础线性代数模块已经学过：
嵌入向量在训练之后会自动组织成有语义结构的空间——
相似的词在空间中距离近，不相似的词距离远。

这个结构不是人为设计的，是训练自然产生的。

当模型被训练来预测下一个词时，
为了准确预测，它需要让语义相关的词有相似的表示——
因为出现在相似上下文中的词（"猫"和"狗"经常出现在类似的句式中）
需要被类似地处理。

### 嵌入的维度

为什么是768维或4096维？为什么不是10维或100万维？

- 维度太低：表达能力不足，无法区分几万个token的语义差异
- 维度太高：参数量和计算量过大，且容易过拟合

实际的维度选择是工程上的权衡。
经验上，模型越大，嵌入维度越高：
- GPT-2：768维
- GPT-3：12288维
- Llama 2 7B：4096维
- Llama 2 70B：8192维

## 从输入到进入Transformer

总结一下文字变成Transformer输入的完整过程：
用户输入："今天天气很好"

步骤1：分词
"今天天气很好" → ["今天", "天气", "很", "好"]

步骤2：Token ID映射
["今天", "天气", "很", "好"] → [4523, 8901, 2345, 7890]

步骤3：嵌入查找
[4523, 8901, 2345, 7890] → [[0.23, -0.87, ...], [1.02, 0.34, ...], ...]
得到4个768维的向量

步骤4：加上位置编码
每个向量加上对应位置的位置编码向量

步骤5：送入Transformer的第一层

**这是模型处理你输入的第一步。后面的所有计算都基于这些向量进行。**

## 关键认知

1. 分词把文字切成子词——介于字符和完整词之间的平衡
2. BPE是最主流的分词算法——通过统计频率迭代合并
3. 分词方式影响不同语言的效率、数字处理能力、罕见词表示
4. 嵌入矩阵把离散的token ID转换成连续的语义向量
5. 嵌入空间的语义结构是训练自然产生的，不是人为设计的
6. 嵌入维度是表达能力和计算成本之间的权衡
