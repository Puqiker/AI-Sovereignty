# 模型服务

> 模型训练好了，然后呢？
> 它还只是一堆存在磁盘上的参数文件。
> 让用户能通过API或界面使用这个模型——这就是模型服务（Model Serving）。
> 理解模型服务的基本概念，对你后面学习私有部署和成本优化至关重要。

---

## 从参数文件到可用的服务

训练完成后的产出是**模型权重文件**——几十到几百GB的参数数据。

把它变成用户可用的服务，需要经过：
模型权重文件
↓
[加载到GPU显存]
↓
[推理引擎封装] ← 处理输入输出格式转换、KV Cache管理等
↓
[API服务层] ← 接收HTTP请求，返回响应
↓
[负载均衡 / 扩缩容] ← 处理并发、流量管理
↓
用户可用的API或界面

---

## 推理引擎

推理引擎是专门为高效推理设计的软件。它和训练框架（PyTorch等）的区别：

训练框架需要支持前向传播和反向传播，记录计算图，灵活性优先。
推理引擎只需要前向传播，可以做大量的性能优化。

主流的推理引擎：
- **vLLM**：专为大语言模型设计，核心优化是PagedAttention（分页管理KV Cache，大幅提升显存利用率和并发能力）
- **TensorRT-LLM**：NVIDIA的推理优化方案，深度利用GPU硬件特性
- **llama.cpp**：极简的CPU/GPU推理实现，支持量化模型在消费级硬件上运行
- **Ollama**：基于llama.cpp的易用封装，一键运行开源模型

### 关键优化技术

**连续批处理（Continuous Batching）**：
不是等一批请求都完成才处理下一批，
而是有请求完成就立刻插入新请求，GPU始终保持满负荷。

**PagedAttention**：
KV Cache不再要求连续内存，而是像操作系统的内存分页一样管理。
这解决了不同请求的KV Cache长度不同导致的内存碎片问题，
大幅提升同时服务的用户数。

**推测解码（Speculative Decoding）**：
用一个小模型快速"猜测"多个token，然后用大模型验证。
如果猜对了，一步产出多个token，加速生成。

## 服务的核心指标

### 延迟（Latency）

从用户发送请求到收到第一个token的时间。

两个阶段的延迟：
- **首token延迟（Time to First Token, TTFT）**：主要由prefill阶段决定
- **每token延迟（Time per Output Token, TPOT）**：主要由decode阶段决定

### 吞吐量（Throughput）

单位时间内能处理的token总数。

吞吐量和延迟往往有权衡——
增大批处理大小可以提高吞吐量，但可能增加单个请求的延迟。

### 并发用户数

同时服务多少用户。
主要受限于GPU显存——每个并发用户需要独立的KV Cache空间。

## 部署方式

### 云API

使用OpenAI、Anthropic等公司的API。

- 优点：零运维、按量付费、始终使用最新模型
- 缺点：数据离开你的控制、成本随用量增长、依赖第三方可用性
- 适合：快速原型、非敏感数据场景

### 自托管开源模型

在自己的服务器或云GPU上部署开源模型。

- 优点：数据不出域、完全控制、可以微调定制
- 缺点：需要运维能力、硬件成本、需要自己优化性能
- 适合：数据敏感场景、需要定制化的企业

### 边缘部署

在用户设备上（手机、PC）运行量化后的小模型。

- 优点：完全离线、零延迟、极致隐私
- 缺点：模型能力受限于设备算力和内存
- 适合：隐私要求极高、需要离线运行的场景

---

## 关键认知

1. 模型服务 = 把训练好的参数文件变成用户可用的API
2. 推理引擎通过连续批处理、PagedAttention等技术大幅提升推理效率
3. TTFT和TPOT是用户体验的两个关键延迟指标
4. 部署方式的选择（云API / 自托管 / 边缘）取决于数据敏感度、成本和能力需求
5. 理解这些让你能评估不同部署方案的优劣，为后面的私有部署专题做准备
