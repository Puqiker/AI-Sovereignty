# 评估方法论

> 你已经知道了模型怎么学习。
> 这个模块讨论一个同样重要但经常被忽视的问题：
> 你怎么知道模型学得好不好？
> "准确率95%"这个数字看起来很漂亮，
> 但它可能在掩盖严重的问题。

---

## 为什么评估比你想的复杂

一个看起来很简单的数字——"准确率95%"——可能意味着：

**场景一**：一个垃圾邮件检测器。
准确率95%。
但实际上95%的邮件本来就不是垃圾邮件。
如果模型把所有邮件都判断为"不是垃圾邮件"，准确率也是95%。
这个模型实际上什么都没学到。

**场景二**：一个疾病诊断系统。
准确率95%。
但在真正患病的100个人中，它只检测出了50个（漏诊了50个）。
那5%的错误全部集中在最需要被正确判断的人身上。

**场景三**：一个语言模型的benchmark测试。
在某个标准测试集上得分95%。
但这个测试集的题目类型和你的实际使用场景完全不同。
模型在你的场景下可能只有60%的表现。

**评估的核心问题不是"分数是多少"，而是"这个分数在你的使用场景下意味着什么"。**

---

## 分类问题的评估：准确率之外的指标

### 混淆矩阵

对于一个二分类问题，所有预测结果可以分为四种：

                模型预测
              正例    负例

实际 正例 TP FN
负例 FP TN

TP = True Positive：真阳性，实际是正例，预测也是正例（正确）
FN = False Negative：假阴性，实际是正例，预测为负例（漏检）
FP = False Positive：假阳性，实际是负例，预测为正例（误报）
TN = True Negative：真阴性，实际是负例，预测也是负例（正确）


### 精确率（Precision）

在所有模型预测为正例的样本中，有多少是真的正例？
Precision = TP / (TP + FP)

高精确率意味着：模型说"是"的时候，大概率真的"是"。
用于场景：你不希望误报太多。比如垃圾邮件过滤——把正常邮件标为垃圾邮件很糟糕。

### 召回率（Recall）

在所有真正的正例中，有多少被模型正确识别出来了？

Recall = TP / (TP + FN)

高召回率意味着：真正的正例很少被遗漏。
用于场景：你不希望漏检太多。比如疾病筛查——漏诊患者比误报健康人更危险。

### F1分数

精确率和召回率往往是矛盾的——提高一个往往会降低另一个。
F1是它们的调和平均，用于在两者之间找平衡：

F1 = 2 × (Precision × Recall) / (Precision + Recall)


### 为什么这些指标很重要

单看准确率会掩盖问题。
你需要根据你的具体场景，决定哪个指标更重要：

- 安全系统（恶意攻击检测）→ 召回率更重要（不能漏掉攻击）
- 内容推荐系统 → 精确率更重要（推荐的内容应该是用户真的想看的）
- 通用场景 → 用F1在精确率和召回率之间取平衡

## 语言模型的评估：更加复杂

语言模型不是简单的分类器。它生成的是自由形式的文本。
评估"一段文本好不好"比评估"分类对不对"困难得多。

### 困惑度（Perplexity）

你在信息论模块已经学过——困惑度衡量模型对测试数据的预测能力。
困惑度越低，模型越好。

困惑度的局限：它衡量的是模型在预测下一个词方面有多好，
但一个困惑度低的模型不一定能生成有用、安全、符合人类偏好的回答。
它只告诉你"模型对语言模式的捕捉能力"，不告诉你"模型对人类有多有用"。

### Benchmark测试

AI社区有大量标准化测试集来评估模型能力：
- MMLU：多学科知识问答
- HumanEval：代码生成能力
- GSM8K：数学推理能力
- TruthfulQA：事实准确性
- HellaSwag：常识推理

这些benchmark的价值在于提供了一个标准化的比较基准。
但它们有严重的局限：

1. **Benchmark泄漏**：如果测试题出现在训练数据中，模型得高分不代表它真的"懂"
2. **分布不匹配**：benchmark测试的问题类型可能和你的实际使用场景完全不同
3. **刷分优化**：模型可以被专门优化来在特定benchmark上得高分，但通用能力并没有提升
4. **饱和效应**：当所有模型在某个benchmark上都得95+分时，它就无法区分模型之间的差异了

### 人类评估

让人类直接评判模型输出的质量。
这是最接近"模型到底有没有用"的评估方式。

但人类评估有自己的问题：
- 昂贵且耗时
- 主观性强——不同评估者可能给出不同判断
- 不可大规模重复

### 基于模型的评估

用一个强模型来评估另一个模型的输出。
比如用GPT-4来评分其他模型的回答质量。

这种方法快速且可扩展，但引入了一个根本问题：
**你在用一个你也没完全理解的系统来评估另一个系统。**

## 你自己的评估能力

在上面这些方法之上，还有一个最重要的评估层：**你自己的判断力。**

当你面对一个具体的问题时，你应该能够：

### 1. 为自己的场景设计评估方法

不要完全依赖别人的benchmark。

问自己：
- 我的使用场景是什么？
- 在我的场景中，什么样的模型表现算"好"？
- 我能不能构建一个代表我实际需求的测试集？

比如你要用AI来辅助代码审查，
那你应该收集一批有安全漏洞的代码片段，
看模型能不能发现这些漏洞——这比任何通用benchmark都更有价值。

### 2. 识别模型的失败模式

每个模型都有它特别容易失败的场景。

不要只看平均性能，要主动寻找：
- 模型在什么类型的输入上表现最差？
- 有没有某一类问题模型几乎总是答错？
- 模型的错误有没有某种系统性的模式？（比如总是在数学推理的某一步出错）

了解失败模式比了解平均准确率更有实际价值。

### 3. 判断评估结果的可信度

当你看到一个评测报告时，问自己：
- 测试集是怎么构建的？是否有泄漏风险？
- 评估指标是否和我关心的能力对应？
- 样本量是否足够大？
- 评估是否在多种场景下进行，还是只在特定场景下？

**不要盲信评测报告。** 任何评测都有其局限性和潜在偏差。

## 过拟合评估本身

一个容易被忽视的陷阱：**你可以对评估方法本身过拟合。**

如果你反复用同一个测试集来调整模型或选择模型，
你实际上在把测试集的信息泄漏进了你的决策过程。
最终选出来的模型可能在这个测试集上表现好，
但在真正的新数据上没那么好。

避免这个问题的做法：
- 测试集只用于最终评估，不用于中间的模型选择（中间用验证集）
- 如果你需要反复评估，定期更新你的测试集
- 意识到任何评估结果都只是对真实性能的一个估计，不是精确值

---

## 关键认知

1. 准确率是一个容易误导的指标——你需要根据场景选择合适的评估指标
2. 精确率和召回率衡量的是不同维度的性能，它们之间往往存在权衡
3. 语言模型的评估远比分类问题复杂——困惑度、benchmark、人类评估各有局限
4. Benchmark分数不等于实际使用价值——分布不匹配、数据泄漏、刷分优化都会扭曲结果
5. 最重要的评估能力是：为自己的场景设计评估方法、识别失败模式、判断评估结果的可信度
6. 评估方法本身也可以被"过拟合"——要警惕测试集信息泄漏
